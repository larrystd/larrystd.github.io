C++ template的最初动机是建立类型安全type-safe的容器如vector, 但之后泛型编程generic programming写出的代码更好。C++ template机制是一个完整的图灵机Turing-complete, 可以用来计算任何可计算的值, 于是有了模板元编程template metaprogramming

条款41 了解隐式接口和编译期多态

多态我认为是对象可以呈现出多种合法类型。C语言中广泛存在的是void指针, 因为void指针能指向任意类型的数据, 因此void指针作为形参或者返回值可以表示任何指针类型。

而在C++对象语义中, 多态表示一个对象可以有多种类型。常见的自然是在继承体系中, 子对象可以表示子类型, 也可以表示成父类型。析构函数常常设置成virtual的, 因为子对象表示成父类型时析构时也要根据子类大小进行析构。而如果某函数形参为父类型, 实际上父对象和子对象都可以输入。这也是运行期多态。

而编译期多态的思路也很简单, 对于泛型T可以接受几乎类型, T这里就表现得多态。

这就是编译期多态, 和运行期多态区别是函数实际类型编译期就已经确定了。而且它得约束时, 能够使函数正常执行。也就是提供.size(),.swap()成员。

条款43 处理好模板基类域的名称

在使用继承时, 如果编译期间有足够信息决定具体类型, 就可以使用基于template的解法。

以上, MsgSender表示信息传输, 而LoggingMsgSender继承之, 表示将信息写入log, 如上LoggingMsgSender调用父类的sendClear, 但这里编译期会报错sendClear不存在, 原因是模板实例化早于继承结构的编译, 模板实例化阶段并不知道Company class信息, 自然不知道有个sendClear函数。这从某种意义上, 从Object Orient C++跨入Template C++似乎不是那么畅通无阻了。

解决办法是写明函数的所有权, 具体有三种

  1. 加上this->表名函数的所有权

  2. 使用using说明函数位于的区域

  3. 写明调用区域是父类的函数。但注意这种办法不适于虚函数, 因为会关闭虚函数绑定。

条款44 将与参数无关的代码抽离

Template是节省时间和避免代码重复的巧妙方法, 但是使用template可能导致代码膨胀, 编译成的二进制码可能带着几乎重复的代码和数据。结果是源码看起来整齐简洁, 但目标码object code却很臃肿。

重复代码的来源主要是模板参数, 因此尽量精简模板参数。

如果使用SquareMatrix<double, 5> sm1;, SquareMatrix<double, 10> sm2;将会实例化出两份代码, 这是template导致代码膨胀的例子。

一个可行的办法是使用继承或者实现将类功能转移, 转移类是不会代码膨胀的。

以上SquareMatrix的执行实际上调用父类的invert实现的, 尽管SquareMatrix被具现化为两个, 但父类SquareMatrixBase不会。同样的对于指针如char*, int*, 应该让这些调用共同的void*的函数来完成实际工作。

条款45 运用成员函数模板接受所有兼容类型

这一条款从智能指针谈起, 智能指针Smart pointers是行为像指针的对象，也是基于模板进行编写。然而真实指针做得很好的一件事是, 支持隐式转换。包括Derive类型转为Base的隐式转换, 但Base类型转为Derive则不被允许。

如上由于Middle继承自Top, 因此在普通指针对象中Middle指针类可以隐式转换为Top指针类。但在模板类中, 产生出的具现体不带有base-derived关系，所以编译器视Smart<middle>和Smart<Top>为完全不同的class。因此想获得隐式转换能力，必须明确写出(因为template几乎不存在隐式转换, int double也不能转)

解决办法是用一个泛化copy构造函数, 这个泛化copy构造函数可以将任意U*类型转换为T*类型, 而转换的要求是满足隐式转换类型规则(例如U*为derive*, T*为Base*可以转; U*为int*, T*为double*则不可以)。.get()函数是取出指针对象smallPtr的内部裸指针。

注意加了泛化copy构造函数还是要定义copy构造函数, 尽管定义泛化copy构造函数但如果不定义copy构造函数编译器还是会生成一个。且当U类型等于T类型时将调用copy构造函数而不是泛化构造函数, 彻底控制template class的copy 构造, 需要结合正常copy构造函数和泛化copy构造函数。

条款46 需要类型转换时为模板定义非成员函数

前面条款24中说明若所有参数皆需类型转换, 为此采用non-member函数。这是为了支持隐式转换对象表达式的正确性。但在模板中仅有这一条是不够的。

问题在于推导T的过程, 根据OneHalf可以推导出T为int; 但根据2却很难推导出T, 原因是template实参推导过程中不将隐式类型转换纳入考虑。

然而一个事实是template实参推导之应用于function template上, 但class template不依赖于template类型推导。将function template转为 class template, 利用friend将function声明为class的friend函数即可。这样在类型推导过程中, class Ration<int>先被具现了出来, friend operator*也被自动声明出来, 后者这时是一个函数而不是函数模板, 编译器因此可以使用隐式转换函数

以上代码能通过编译, 但不能连接。因为friend声明的函数位于Rational对象内, 不能被外界的operator*看到。可行的办法是将外界实现放入到内部

此外, 外部函数虽然看不Rational内部定义,但Rational内部可以看到外部函数, 因此可以让内部调用外部函数

值得注意的是这一条还是将模板类在不支持隐式转换条件下怎样模拟类似隐式转换的功能, 事实上类应该尽量减少隐式转换, 不支持隐式转换。

条款47 请使用traits classes表现类型信息

STL主要由用以表现容器, 迭代器和算法的template组成。STL共有五种迭代器,

  1. Input迭代器只能向前移动, 一次一步, 客户只可读取不能写入, 且只能读取一次
  2. Output迭代器和Input类似, 但只能输出。一次一步, 只能输出一次。
  3. forward迭代器可以向前读写,一次一步, 读写多次。类似于单向链表slist的移动。
  4. Bidirectional迭代器可以向前向后移动1步, 类似于双向链表list移动, 以及set, map的迭代器。
  5. random access迭代器可以前后移动任意距离(通过迭代器加减), vector, deque, string提供的迭代器是这一步。

五个迭代器具有继承关系, 有效的is-a

Trait则是一种技术, 输入迭代器类型作为模板参数, 可以得到迭代器的实际类型。即使迭代器为指针, 也能得到指针指向元素的类型。注意输入模板参数, 因此这是编译期判断的迭代器指向对象的类型, (对于指针来说, 运行期难以判断其指向对象的类型)

条款48 认识template 元编程

Template metaprogramming TMP模板元编程, 是编写template-based C++程序并执行于编译期的过程。使用TMP的C++程序可能在每个方面都更高效, 较小的可执行文件, 较短额运行期, 较少的内存需求。但编译时间变长了, 且难以调试, 不能动态执行(执行的是编译好的程序, 不可能存在来一个连接就分配一块内存的动态形式)。

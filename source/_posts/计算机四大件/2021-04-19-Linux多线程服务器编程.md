### 析构函数遇到多线程

当一个对象能够被多线程看到时，那么对象的销毁时机就会变得模糊不清，可能出现多种竞态条件(race condition)
* 即将析构一个对象，别的线程正在执行该对象的成员函数
* 保证在执行成员函数期间，对象可能被其他线程析构
* 调用成员函数之前，需要确保该对象还活着，且不会被执行析构。

以上困难均可以基于`shared_ptr`解决。注意到倘若对象动态创建用指针访问，即便函数加了锁是线程安全的，其对象销毁的race condition仍然会存在。

### 对象的创建
对象的构造做到线程安全，要求是构造期间不要泄漏`this`指针，即
* 不要在构造函数中注册回调
* 不要在构造函数中把`this`传给跨线程的对象
如果`this`指针被泄漏给其他对象，可能会在多线程中调用尚未构造完的当前对象。

```cpp
// 不要这样做
class Foo : public Observer
{
    public:
    Foo(Observer* s) {
        s->register_(this); // 非线程安全
    }

    virtual void update();
};

// 要这样做
class Foo : public Observer {
    public:
    Foo();
    virtual void update();

// 另外定义一个函数，构造之后执行回调工作
    void observe(Observable* s) {
        s->register_(this);
    }
};

Foo* pFoo = new Foo;
Observable* s = getSubject();
pFoo->observe(s);   // 二段式构造
```

我感觉对于对象的构造和析构，都使用单线程进行比较好，对于计算密集或者多连接，再使用多线程。

### mutex难以保护析构函数

多线程时用`mutex`保护临界区，但某个线程执行析构函数将`mutex`销毁，那么其他持有`mutex`的线程将会出现未知的错误。 因而由于`mutex`可能被析构销毁，因此不能保护多线程中的析构函数。

### 线程安全的Observer有多难
观察者模式Observer，定义多个对象存在的一对多的依赖关系，当一个对象状态发生变化，所以依赖它的对象都得到通知并被自动更新。又称发布-订阅模式。

面向对象程序设计中，对象的关系主要三种: `composition、aggregation、assocation`。 composition(组合)关系没有太多麻烦，在于对象生命期由唯一owner控制,owner析构时会把x也析构掉。   

aggregation(聚合)和assocation(关联)表示对象a用到了对象b，但只是持有其指针，b的生命期不由a控制。聚合关系相比关联关系只是多了a和b 有逻辑上的整体和部分关系。   

聚合和关联关系面临的重要问题是判断当前对象是否存活，注意很难通过指针判断对象当前是否存活。  Java没有这个问题，因为Java中一个reference只要不为null，它一定指向有效的对象。    

例如在Observer模式，也就是行为模式中。

```cpp
class Observer {
public:
    virtual ~Observer();
    virtual void update() = 0;
    // ...
};

class Observable {
public:
    void register_(Observer* x);
    void unregister(Observer* x);

    void notifyObservers() {
        for (Observer* x : observers_) {
            x->update();    // 如何判断x指向的对象存活?
        }
    }
private:
    std::vector<Observer*> observers_;
};

```
存在的问题,1. `x->update()`如何判断对象活着。 2. `x->update()`时其他线程正在析构本对象。


解决以上两个问题的方法是使用智能指针。

1. shared_ptr控制对象的生命期，shared_ptr是强引用，只要有指向对象的shared_ptr存在，该对象就不会被析构。当指向x的最后一个`shared_ptr`析构或`reset()`的时候，x保证被销毁。**这解决了多线程析构问题**。

2. weak_ptr不控制对象的生命期，但它知道对象是否还活着，如果对象活着，它可以提升为有效的shared_ptr，如果对象死亡，提升会失败，返回空的shared_ptr。**这解决了对象是否存活问题**

3. `shared_ptr`/`weak_ptr`的计数在主流平台上是原子操作，没有用锁，性能不俗。 `shared_ptr`/`weak_ptr`的线程安全级别与`std::string`和STL容器一样。

将以上方法应用到Observer上,利用`weak_ptr`探查对象的生死
```cpp
class Observable
{
    public:
    void register_(weak_ptr<Observer> x)
    void notifyObservers();

    private:
    mutable MutexLock mutex_;
    std::vector<weak_ptr<Observer>> observers_;
    typedef std::vector<weak_ptr<Observer>>::iterator Iterator;
};

void Observable::notifyObservers(){
    MutexLockGuard lock(mutex_);    // 互斥锁
    Iterator it = observers_.begin();
    while (it != observer_.end()){
        shared_ptr<Observer> obj(it->lock());   // 尝试提升，这一步线程安全

        if (obj){   // 提升成功，说明对象存活
            obj->update();
            ++it;
        }else{
            it = observers_.erase(it);  // 对象已经销毁，拿掉该对象指针
        }
    }
}
```
### C++内存问题和解决

C++可能出现的内存问题
1. 缓冲区溢出(buffer overflow)
2. 空悬指针/野指针
3. 重复释放(double delete)
4. 内存泄漏(memory leak)
5. 不配对的new/delete
6. 内存碎片 (memory fragmentation)

正确使用智能指针可以解决前五个问题。
1. 缓冲区溢出。用`std::vector<char>`/`std::string`或自己编写class管理缓冲区，记住缓冲区长度，通过成员函数而不是裸指针访问缓冲区。
2. 空悬指针/野指针。用`shared_ptr`/`weak_ptr`。
3. 重复释放。用`scoped_ptr`，只在对象析构时候释放一次。
4. 内存泄漏。用`scoped_ptr`，对象析构时候自动释放内存
5. 不配对的new/delete。替换成std::vector/scoped_array。




### shared_ptr的问题
如果在某个类中定义了智能指针shared_ptr，并且该指针指向了某对象资源，则该对象的释放工作将在该类析构函数调用后自动进行。
注意`shared_ptr`引用计数是安全且无锁的，但对象的读写不是。`shared_ptr`的线程安全级别和内建类型、标准库容器、std::string一样。即
* 一个shared_ptr对象实体可被多个线程同时读取;
* 两个shared_ptr对象实体可被两个线程同时写入;
* 多个线程读写同一个`shared_ptr`对象，需要加锁。

另外，shared_ptr/weak_ptr均为值语义的对象。

```cpp
void read() {
    shared_ptr<Foo> localPtr;   // 创建对象在临界区外
    {
        MutexLockGuard lock(mutex);
        localPtr = globalPtr;   // read 需要加锁
    }
}
```

`shared_ptr`的简单使用
```cpp
int main()
{
    std::shared_ptr<Person> p1(new Person(1));// Person(1)的引用计数为1

    std::shared_ptr<Person> p2 = std::make_shared<Person>(2);

    p1.reset(new Person(3));// 首先生成新对象，然后引用计数减1，引用计数为0，故析构Person(1)
                            // 最后将新对象的指针交给智能指针
                            // 其实相当于指针指向新对象

    std::shared_ptr<Person> p3 = p1;//现在p1和p3同时指向Person(3)，Person(3)的引用计数为2

    p1.reset();//Person(3)的引用计数为1
    p3.reset();//Person(3)的引用计数为0，析构Person(3)
    return 0;
}
// shared_ptr的reset( )函数的作用是将引用计数减1,停止对指针的共享,但除非引用计数为0,否则不会发生删除操作。
//输出
Construct 1
Construct 2
Construct 3
Destruct 1
Destruct 3
Destruct 2
```

### 对象池

先膜拜一下，下面的东西太牛批了，陈硕真滴大佬啊。

列出一个需求，假设有`Stock`类，代表股票的价格，并且有唯一的标识。为了节省资源，同一个程序里面每一只出现的股票只有一个Stock对象，如果多处用同一只股票则应该被共享，如果没有再用到则应该被析构（这隐含了引用计数）。
这个案例隐含着，股票一旦被创建就会被共享使用，一旦某时刻无人使用该对象，对象就会析构股票对象，不会等着被使用，可能与现实有出入。

```cpp
// code version 1
class StockFactory : boost::noncopyable {
public:
    shared_ptr<Stock> get(const string& key);
private:
    mutable MutexLock mutex_;
    std::map<string, weak_ptr<Stock> > stocks_; // 如果weak_ptr改为shared_ptr，则在创建stocks_时Stock自动获取一个引用计数，这导致Stock对象将不会被销毁，生命期与 StockFactory相同
};

shared_ptr<Stock> StockFactory::get(const string& key) {
    shared_ptr<Stock> pStock;
    MutexLockGuard lock(mutex_);
    weak_ptr<Stock>& wkStock = stocks_[key];   
    pStock = wkStock.lock();    // 尝试提升，判断对象是否存活
    if (!pStock) {
        // reset，当前对象计数减1（可能被销毁）,指向新的对象计数加1，也就是Stock(key)
        pStock.reset(new Stock(key));
        // wkStock是一个引用
        wkStock = pStock();
    }
    return pStock;
}
```

以上代码通过` weak_ptr `，`reset`解决了Stock对象的销毁问题。若`stocks_`map容器包含`shared_ptr`对象，则创建stocks_时Stock自动获取一个引用计数，这导致Stock对象将不会被销毁（除非在别处刻意调用reset()，但这个不可控。

但以上做法还有一个问题，是`Stock`对象虽然可以销毁，但`stocks_`却只增不减。前者销毁只是让stocks_部分元素变为`NULL`而已。 这时候可以使用`shared_ptr`的`reset`定制析构功能。

```cpp
class StockFactory : boost::noncopyable {
public:
    shared_ptr<Stock> get(const string& key);
private:
    mutable MutexLock mutex_;
    std::map<string, weak_ptr<Stock> > stocks_; // 如果weak_ptr改为shared_ptr，则在创建stocks_时Stock自动获取一个引用计数，这导致Stock对象将不会被销毁，生命期与 StockFactory相同
};

shared_ptr<Stock> StockFactory::get(const string& key) {
    shared_ptr<Stock> pStock;
    MutexLockGuard lock(mutex_);
    weak_ptr<Stock>& wkStock = stocks_[key];   
    pStock = wkStock.lock();    // 尝试提升，判断对象是否存活
    if (!pStock) {
        // reset，当前对象计数减1（可能被销毁）,指向新的对象计数加1，也就是Stock(key)
        pStock.reset(new Stock(key),
        boost::bind(&StockFactory::deleteStock, this, _1));
        // wkStock是一个引用
        wkStock = pStock();
    }
    return pStock;
}

void deleteStock(Stock* stock) {
    if (stock) {
        MutexLockGuard lock(mutex);
        stocks_.erase(stock->key());
    }
    delete stock;
}
```
以上有个安全问题，
然而，` boost::bind(&StockFactory::deleteStock, this, _1))`传入的this指针，这可能出现`StockFactory`对象先于`Stock`对象析构，这样会core dump。我们需要确保`StockFactory`对象生存期在`Stock`对象之后。注意这两个对象是由`shared_ptr`管理生存期而非手动`delete`，且这两个对象不互相依赖，完全有可能`StockFactory`引用为0先于`Stock`析构。

这时候将传入的this指针用shared_from_this(this)改为`boost::bind(&StockFactory::deleteStock, shared_from_this(), _1));`就可以让`StockFactory`获得一个`shared_ptr<StockFactory> ` ，保证引用计数至少为1，调用`StockFactory::deleteStock`时`StockFactory`对象不会被析构。

上述使用`shared_from_this()`可以保证调用`StockFactory::deleteStock`时`StockFactory`对象一直存在，但也同时延长了对象生命期。 我们往往需要 如果对象活着就调用其成员函数，否则忽略之的语义，也就是弱回调。

以下是完整的代码
```cpp
class StockFactory : boost::noncopyable {
    shared_ptr<Stock> get(const string& key) {
        shared_ptr<Stock> pStock;
        MutexLockGuard lock(mutex_);
        weak_ptr<Stock>& wkStock = stocks_[key];   
        pStock = wkStock.lock();    // 尝试提升，判断对象是否存活
        if (!pStock) {
            // reset，当前对象计数减1（可能被销毁）,指向新的对象计数加1，也就是Stock(key)
            pStock.reset(new Stock(key),
            boost::bind(&StockFacory::weakDeleteCallback,
        // 将shared_ptr转换为weak_ptr，不延长StockFactory生命期
            boost::weak_ptr<StockFactory>(shared_from_this()),
            _1));
            // wkStock是一个引用
            wkStock = pStock();
            }
        return pStock;
    }

    void weakDeleteStock(const boost::weak_ptr<StockFactory>& wkFactory, 
    Stock* stock) {
        shared_ptr<StockFactory> factory(wkFactory.lock()); // 尝试提升,StockFactory是否存在
        if (factory) {
            factory.removeStock(stock);
        }
        delete stock;
    }

    void removeStock(Stock* stock) {
        if (stock) {
            MutexLockGuard lock(mutex_);
            stocks_.erase(stock->key());    // 从stocks_中擦去要删除的Stock对
        }
    }
private:
    mutable MutexLockGuard lock(mutex_);
    std::map<string, weak_ptr<Stock> > stocks_;
};
```

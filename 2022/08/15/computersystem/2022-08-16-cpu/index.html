流水线机制

CPU 的流水线机制, CPU Instruction pipelining。流水线机制一个重要的特性就是 提高了系统的吞吐量

流水线针对的指令多个阶段的并行执行, 如下I1, I2, I3是指令, A,B,C是指令运行的三个阶段, 横轴是时钟周期。指令的阶段包含取指(InstrucTIon Fetch), 译码(InstrucTIon Decode), 执行(InstrucTIon Execute), 访存(Memory Access), 写回(Write-Back), 而流水线也就是针对这些阶段, 不同指令的不同阶段是可以并行执行的。也就是针对cpu而言, 一个指令访存时它可以执行其他指令

乱序执行

一个指令的执行被分成：取指、译码、访存、执行、写回、等若干个阶段。然后，多条指令可以同时存在于流水线中，同时被执行。指令流水线并不是串行的，并不会因为一个耗时很长的指令在”执行”阶段呆很长时间，而导致后续的指令都卡在”执行”之前的阶段上。(如果一个指令执行很长时间, 在时间片允许内, CPU可以要求其他指令去执行访存等其他操作)、

同时, 不同执行指令也可能并行。CPU有一个加法器和一个除法器，那么一条加法指令和一条除法指令就可能同时处于”执行”阶段, 而两条加法指令在”执行”阶段就只能串行工作。

针对程序而言, 编译之后是多条指令, 这些指令被流水线同时执行可能带来乱序。比如一条加法指令原本出现在一条除法指令的后面，但是由于除法的执行时间很长，在它执行完之前，加法可能先执行完了。再比如两条访存指令，可能由于第二条指令命中了cache而导致它先于第一条指令完成。

因而CPU乱序需要保证程序上下文因果顺序,

由于b=f(a)这条指令依赖于前一条指令a++的执行结果，所以b=f(a)将在执行阶段之前被阻塞，直到a++的执行结果被生成出来；而c–跟前面没有依赖。因此编译优化的手段可以是则试图通过指令重排将这样的两条指令拉开距离, 以至于后一条指令进入CPU的时候，前一条指令结果已经得到了

从指令乱序角度分析编译优化更直接了

指令重排序

指令重排序不管在单线程还是多线程, 需要遵循的准则就是happen-before因果顺序准则。单线程编译器保证了编译优化和指令乱序执行保证之, 多线程需要通过锁, 内存屏障, 原子类等保证

编译期重排序的典型就是通过调整指令顺序，在不改变程序语义的前提下，尽可能减少寄存器的读取、存储次数，充分复用寄存器的存储值(别频繁读写寄存器)。

单线程下指令执行的值可以暂存到cache中, 多线程情况下不一定命中上一个线程的结果。局部阻止重排序的发生，例如对volatile变量的操作指令都不会被重排序，因为如果重排序，又可能产生可见性问题。

分支预测

StackOverflow 上一个非常有名的问题—— Why is processing a sorted array faster than processing an unsorted array? 对一个数组中的每个元素，先做判断，如果大于某个值，就做累加，就是这样一个简单的操作，发现一个有意思的现象，如果用 C++ 写这段代码，对于有序数组和无序数组分别做这个操作，性能大概相差五倍多，在 Java 中，差距小一点，大概是 1 倍。为什么会出现这个问题呢？

背后的原因就是 CPU 流水线下，CPU 采用分支预测技术，对于有序数组可以很好地 CPU 这一特性，而无序数组会使得分支预测手足无措。

如果没有流水线机制，一条指令大概会花费 4 个时钟周期，而如果采用流水线机制，当第一条指令完成Fetch后，第二条指令就可以进行Fetch了，极大提高了指令的执行效率。

流水线的指令1, 指令2, 指令3往往是乱序执行的。但对于条件指令, 需要要求判断条件指令执行完后才能执行下面的指令。条件指令的这种因果关系对于乱序执行的流水线指令是不友好的。例如可能指令1的Execute时间过长这样指令2的Execute就无法执行

分支预测技术

如果没有分支预测，那么CPU需要等待条件执行完后才能知道下一步的跳转地址。为了消除之间的等待时间，CPU会去预测其条件结果，是更倾向于true，还是更倾向于false。在预测错误的情况下，需要忽略之后的指令，并且重新加载正确的指令。重新加载指令所造成的时间浪费取决于流水线架构CPU的层级。现代CPU往往拥有非常大的流水线级别，很容易造成10~20时钟周期的浪费。一个时钟周期一般不超过10ns

静态分支预测是一个非常简单的策略，不依赖执行环境和执行过程中的状态。比如某些CPU，总是会预测一个条件是false的，有些则会对向前条件和向后条件做不同的预测，甚至有些架构允许插入预测代码。但总的来说，静态预测永远是编译器就已经做好了。

动态分支预测则是在运行期间来决定我们的预测结果。一般对于结果的预测都是依赖于历史结果，也就是该条件前几次的执行结果进行统计。

  1. 显然有倾向的条件适合分支预测, 例如返回的status 往往是OK, 只有少数情况status < 0表示出现问题。
  2. 另外在循环中的条件判断, 由于循环局部性, 本次的执行条件会参考上次循环结果, 也常适用于分支预测 分支预测的思路可以是删除条件判断, 使指令可以并行, 这对并行计算也有参考性。例如多线程修改共享变量, 可以让线程先修改, 修改完后对比(修改的对不对), 也就是CAS

指令集

Reduced Instruction Set Computing (RISC) 精简指令集，比如ARM,MIPS

Complex Instruction Set Computing (CISC) 复杂指令集，比如Intel的X86

32位MIPS指令集，共三种形式, R型, I型, J型

J型，让CPU调到一个地址，然后从这个地址开始执行指令

I型指令，它可以将一个任意数(立即数 immediate value)加上寄存器里的数字，然后把结果存储到另外一个寄存器里面:

R型指令，连续三个5位二进制码来表示三个寄存器的地址，然后用一个5位二进制码来表示移位的位数（如果未使用移位操作，则全为0），最后为6位的function码(它与opcode码共同决定R型指令的具体操作方式)

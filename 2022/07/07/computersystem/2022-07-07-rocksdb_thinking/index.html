事务transaction

lsn log sequence number

RocksDB中的每一条记录(KeyValue)都有一个LogSequenceNumber(后面统称lsn)，从最初的0开始，每次写入加1。该值为逻辑量，区别于InnoDB的lsn为redo log物理写入字节量。

snapshot

Snapshot是RocksDB的快照，实际存储的就是一个lsn.

查询时如果设置了snapshot为某个lsn, 那么对于此snapshot的读来说，只能看到lsn(key)<=lsn(snapshot)的key，大于该lsn的key是不可见的。

隔离级别

为了实现事务下的一致性非锁定读(读可以并发)，不同的数据库(引擎)实现了不同的读隔离级别。

  1. Read Uncommited
  2. Read Commited
  3. Repeatable
  4. ReadSerializable

事务并发读的关键在于隔离, 即当前事务写的数据在未提交前不可被其他事务读到。rocksdb基于私有空间writeBatch实现隔离

事务内的写入会暂存在自己的 WriteBatch 中，事务内读取时会先读取自己的 WriteBatch，再读取 MemTable、L0、L1 等。期间其他事务看不到 WriteBatch 中的内容，到事务提交时，WriteBatch 的值会落入 MemTable 和 WAL，才可以使其他事务看到。

乐观并发控制

乐观并发控制事务中，会跟踪事务内 Key 的读写，到 Commit() 时进行冲突检测，检查这些 Key 是否被其他事务修改过了，如果有，则放弃提交写 MemTable，就等于什么都没有发生；

悲观并发控制事务中，会对事务内写入的 Key 进行上锁，保证memtable对应的key不可被其他事务修改(可读), 当 Commit() 中当这些 Key 落盘之后，释放锁，就没有冲突检测环节了；

每次写入 WriteBatch 时会递增 sequence 号，Snapshot 的内容就是这个 sequence 号，查询时过滤按小于等于 sequence 号的 Key 的最新值为准。sequence number也起到隔离的作用。相同值不同sequence的key相当于一串版本链

每次写入 WriteBatch 时会递增 sequence 号，Snapshot 的内容就是这个 sequence 号，其他事务查询时需要在memtable和SST中过滤按小于等于 sequence 号的 Key 的最新值为准。

ACID保证

rocksdb数据库引擎事务类型：OptimisticTransaction（乐观型事务）和PessimisticTransaction（悲观型事务）。

两种类型事务其中差别之一：冲突检测时机，OptimisticTransaction在Commit操作时，通过OptimisticTransactionCallback回调CheckTransactionForConflicts进行冲突检测;

PessimisticTransaction（默认事务类型）在Put、Delete等操作记录时，通过TryLock方法针对tracked key加锁，Commit完成时通过UnLock方法解锁。

各个事务内部维护一个WriteBatch(内存对象)，本事务Commit之前优先读该WriteBatch(同一事务内写操作可见)，写操作直接写入本事务WriteBatch，提交依次写入WAL和MemTable，依赖WriteBatch的原子性和隔离性实现了ACID。

  1. 原子性(A): 事务内一次执行过程所包含各项操作, 全部成功或全部失败; 依赖WriteBatch实现;

  2. 一致性(C): 事务执行不破坏数据完整性, 从一个一致性状态到另一个一致性状态; 依赖WAL和WriteBatch实现;

  3. 隔离性(I)：并发事务相互隔离，事务之间不相互干扰；依赖并发控制、MVVC实现

  4. 持久性(D)：事务提交后对于记录更改操作永久保存；依赖SST持久化文件、WAL实现

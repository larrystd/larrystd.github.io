### 协程

对于单核CPU, 从硬件角度上某个时刻只能运行一个进程(或者线程)。但从操作系统层面是以多进程为运行的基本形式, 换句话说, 某时刻系统存在大量进程(但每个时刻只能运行一个), 调度这些进程就是操作系统的任务之一。由于时间片的存在，我们写的多线程程序相比单线程, 优势就是增加了用户程序的执行时间, 从单线程的一个时间片变成了多个时间片。

那问题来了，如果我们在计算机上只运行一个线程，这个单线程模型是否比多线程效率更高。理论下是的，但有两个问题。首先我们的操作系统不可能存在只运行一个线程，用户线程还需要与大量的系统守护线程争夺时间片。另外一个是IO问题, 也就是这个线程读写磁盘时必须亲自去干，这会耗费大量时间在等待磁头旋转上面，而多线程就可以分工。因此，选择多线程在实践上往往优于单线程。

但是多线程问题是切换上下文开销的问题，如果线程数量几十上百，这个时间是可观的。此外线程的目的是令用户程序获得更多时间片执行(多线程映射到同一块代码段), 但线程数量的增加往往会边际成本增加，效益减少，因此我们要使用多线程但不要使用太多的线程。

虽然IO多路复用实现了少量线程监听大量文件描述符，但是对于更多的小逻辑，我们不能轻易通过开线程解决(线程不宜太多，需要用到刀刃上)，对于这些小逻辑可能也没有多路复用机制，解决办法就是通过协程。但以上看出，协程目的是防止线程太多导致的受益降低，并非没有线程。有人考虑用多进程和协程从而实现所谓的协程替代线程，但linux中线程和进程几乎是一样的，线程也可以享受多核CPU带来的并行优势, 而协程是用户态的，协程和进程、线程没有太多关联。将协程理解成用户态对于系统态进程线程的补充就行了，因为系统态太多进程或线程导致效益降低。

很多编程语言（C#、erlang、golang）都在语言层面实现了协程语义。C/C++本身不支持协程，但存在不少开源协程库。协程的原理很简单，当程序需要等待时(例如IO)使用类似goto的手段执行其他不需要等待的程序, 此期间文件系统线程正在读写文件, 读写完毕后协程再回到等待的位置继续执行。从汇编上理解就是goto, jmt, 协程因此难免造成可读性下降。

C语言的ucontext和C++20的协程是有栈协程和无栈协程的代表.此外, goroutine使用有栈协程, Rust和Js使用无栈协程.
### C语言的ucontext 

通过程序的跳跃执行实现协程特别像汇编的jmt, C语言的uncontext库就是这样实现协程的。协程跳转前需要记录所处的位置环境, ucontext也就是`ucontext_t`结构体。
```cpp
typedef struct ucontext_t
  {
    unsigned long __ctx(uc_flags);  
    struct ucontext_t *uc_link;     // uc_link指向当前context 结束时待恢复的上下文
    stack_t uc_stack;               // context 使用的栈
    sigset_t uc_sigmask;            // context 要阻塞的信号合集
    mcontext_t uc_mcontext;         // 机器特定的 context 表示
  } ucontext_t;
```

#### 四个API
下列程序表示getcontext获取位置, 协程继续执行, 执行到setcontext跳转到getcontext的位置, 这个实现和汇编的loop循环的实现大体相同。
```cpp
int getcontext (ucontext_t *ucp); : 当前协程的context 保存到 ucp 指向的地址。
int setcontext (const ucontext_t *ucp); 设置ucp 指向的context为运行上下文并开始执行, 相当于goto 到getcontext的位置


#include<ucontext.h>
#include<unistd.h>
#include<stdio.h>

int main(void)
{
    ucontext_t context;

    getcontext(&context); // 上面文保存到context
    printf("Hello world\n");
    sleep(1);
    setcontext(&context); // 跳转到getcontext保存的context位置
    return 0;
}
输出一直是hello world
```

协程的主要操作就是, 保存上下文, 执行保存过的上下文, 对应的就是`getcontext`和`setcontext`。除此之外还有两个api为了方便使用, 分别是`makecontext`根据参数创建一个上下文, `swapcontext`保存+执行上下文。

`extern void makecontext (ucontext_t *__ucp, void (*__func) (void), int __argc, ...)`
修改ucp指向的上下文的任务。后面三个参数为func和参数个数, 上下文状态保存在了用户的堆栈，而不是系统层面调用函数的入栈出栈。这种实现协程的办法也称为有栈协程，设置栈的大小是一个技术活。Go语言实现了一个协程栈扩容的机制，相对比较优雅的解决了这个问题。

```cpp
void print_hello()
{
    printf("Hello \n");
}

int main()
{
    ucontext_t context;
    char stack[2048];
    getcontext(&context);   // 需要先利用getcontext初始化
    
    context.uc_stack.ss_sp = stack; // 修改元素, 使用的栈
    context.uc_stack.ss_size = sizeof(stack);
    context.uc_link = NULL; 
    context.uc_stack.ss_flags = 0;
    makecontext(&context, (void (*) (void))print_hello, 0);   // 重新makcontext         
    setcontext(&context);   // 跳跃执行
    return 0;
}

输出
hello
```

`int swapcontext (ucontext_t restrict oucp, const ucontext_t restrict ucp);`将当前的context保存到oucp中，并切换到ucp指向context继续执行。该函数起到调度多个unp执行的作用。
```cpp
#define CR_N 8			/* Number of coroutines */
#define STACK_SZ 2048		/* Stack size */

ucontext_t      mainCtx;	/* main context */
ucontext_t      cr[CR_N];	/* coroutines contexts */

char            stack[CR_N][STACK_SZ];	/* All stacks for all coroutines */

void
func(size_t id)
{
	size_t          nid;

	printf("I'm coroutine %lu\n", id);

	if ((nid = id + 1) < CR_N)
		setcontext(&cr[nid]);	// 从cr[id]切换到cr[id+1]
	else
		setcontext(&mainCtx);	// 切换到main的上下文
}

int
main(void)
{
	size_t          i;

	for (i = 0; i < CR_N; ++i) {	// 创建CR_N个协程上下文context
		getcontext(&cr[i]);

		cr[i].uc_link = NULL;

		cr[i].uc_stack.ss_sp    = stack[i];
		cr[i].uc_stack.ss_flags = 0;
		cr[i].uc_stack.ss_size  = STACK_SZ;

		makecontext(&cr[i], (void (*) (void)) func, 1, i);
	}

	puts("I'm main and I'm going to swap to coroutine 0");
	swapcontext(&mainCtx, &cr[0]);	// 记录当前位置为mainCtx, 并开始执行cr[0]

	puts("completed");

	return EXIT_SUCCESS;
}
输出
I'm main and I'm going to swap to coroutine 0
I'm coroutine 0
I'm coroutine 1
I'm coroutine 2
I'm coroutine 3
I'm coroutine 4
I'm coroutine 5
I'm coroutine 6
I'm coroutine 7
completed
```

#### 有栈协程和无栈协程

实现一个协程的关键点在于如何保存、恢复和切换上下文。已知函数运行在调用栈上；如果将一个函数作为协程，保存上下文即是保存栈帧存储的值，以及此时寄存器存储的值(函数用到的变量在寄存器或者栈上); 恢复上下文即是将这些值分别重新写入对应的栈帧和寄存器；而切换上下文无非是保存当前正在运行的函数的上下文，恢复下一个将要运行的函数的上下文。有栈协程便是这种朴素思想下的产物。相比系统调用, 有栈协程省去了等待系统线程的开销, 但还是要记录现场和恢复现场。

无栈协程采用类似生成器(generator)的思路实现了上下文切换, 在python中, 生成器认为看成用返回为yield的函数。这样由 yield 切割的代码组成一个状态机，而调用next() 就是执行下一个状态。显然这适合相当于有一个状态链表, 通过next就可以执行下一个状态。状态机可以看成一个有向图, 节点类似一个闭包, 能够捕获变量放入一个匿名的结构中。在编译时会确保生成的状态机符合程序运行的流程。

但利用状态机的无栈协程肯定不如有栈的协程自由, 例如无栈的JS不能这样写
```js
async function processArray(array) {
    // 显然这里 forEach 是个嵌套函数
    array.forEach(item => {
        // Uncaught SyntaxError:
        // await is only valid in async function
        const result = await processItem(item)
        ...
    })
}
```
而有栈的go可以
```go
func processArray(array []int) {
    for i := 0; i < len(array); i++ {
        ch := make(chan int)
        go processItem(array[i], ch)
        result := <- ch
        ...
    }
}
```
有栈协程对于旧代码的兼容度也更多, 原因在于无栈协程的状态机结构抽象高, 不符合状态机的流程就不能通过无栈协程执行, 因此不能被非协程函数嵌套调用。有栈协程的代表是Go, 无栈协程的代表是Javascript和Rust。有种来到现代语言的感觉hhhh

额难以置信的是, C++20居然采用无栈协程, 看来C++也要在广大现代语言中踩一脚了。据说C++20是继C++11之后变动最大的版本, 在我看来C++11, 14, 17都在提高C++模板编译能力, 因为编译优化确实是C++核心大杀招, 能大大提高运行效率。额所以协程, 网络库这种还没上, 以后C++标准应该会放在这些上面了。

### C++20的协程
重头戏来了, C++20的协程。和C语言朴素类似go的上下文切换, C++20的协程内部实现逻辑是生成器的无栈协程。同时和C++异步对象Promise结合, 以期达到协程代替异步编程的目的。

#### 异步编程
C++20协程主要用来解决异步编程的麻烦, **异步编程通常是新建一个线程执行一个逻辑, 逻辑执行完后通过回调函数通知主线程**。但问题在于回调通知主线程往往比较麻烦, 且开一个线程需要的资源是不可忽略的。
```cpp
using call_back = std::function<void(int)>;
void AddByCallback(int init, call_back f) //add之后的结果由回调函数f通知
{
    std::thread t([init, f]() { // 开一个线程, 执行逻辑init+100, 然后调用回调函数f
        std::this_thread::sleep_for(std::chrono::seconds(5)); // sleep一下，假装很耗时
        f(init + 100); // 耗时的计算完成了，调用回调函数
    });
    t.detach();
}
```

C++中通用的异步编程是通过promise和future, future用来获取异步线程执行的内容。但是这种办法主线程需要阻塞等待子线程执行完, 也不能解决开一个线程开销的问题。
```cpp
std::promise<int> pr;
// 开一个线程进行异步操作, 异步结果放在promise中, 主线程通过future.get()获取
std::thread t([](std::promise<int>& p){ p.set_value_at_thread_exit(9); },std::ref(pr));//这里promise拿到了9
std::future<int> f = pr.get_future();//设置关联的future
auto r = f.get();   // 得到9

// 或者用async把std::thread和std::promise结合起来
std::future<int> f1 = std::async(std::launch::async, [](){ 
  return 8;  
}); 
cout<<f1.get()<<endl; //output: 8
```
#### 协程关键字的引入

类似于goroutine, C++20实现的协程通过关键字co_await/co_yield/co_return声明函数该被协程执行。协程本身当然也需要线程执行, 只是这个线程执行的逻辑并非代码序, 而是协程规定的顺序。想必执行协程的线程也如go一般有线程池维护。

co_await/co_yield/co_return三个操作符声明了所在函数应该由协程执行, 但有各自的含义。

co_await 说明暂停执行当前函数,(转执行别的函数, 例如IO), 直到恢复(例如IO执行完了恢复上下文执行)。
```cpp
// 下面这个函数被协程执行, 执行到co_await暂停去执行socket.async_read_some直到resume
task<> tcp_echo_server() {
  char data[1024];
  while (true) {
    size_t n = co_await socket.async_read_some(buffer(data));
    co_await async_write(socket, buffer(data, n));
  }
}
```

co_yield, 暂停当前函数执行, 返回值
```cpp
generator<int> iota(int n = 0) {
  while(true)
    co_yield n++;
}
```

co_return, 终止当前函数执行, 返回值
```cpp
lazy<int> f() {
  co_return 7;
}
```

#### 协程和Promise
以上三个关键字如果golang 去执行协程的go关键字, 但C++不是这样就完了。它没有提供golang那样完整的协程支持, 让你一个go 就执行协程不用管了。C++20中你还得设置协程内部的调度规则, 例如什么时候唤醒suspend的协程, 甚至协程创建前后的初始化你也要写。。。

Promise规定了协程的状态, 例如协程切换时要保存上下文, 这可以通过栈或者状态机的方式。C++的无栈协程是通过一个状态, 额没错Promise中你要定义这个状态对象。如果你的关键字是co_return会返回值, 那返回前可以Promise的final_suspend()处理返回值。

基于协程的好处就是以同步方式写异步代码，就没有callback地狱了。这时候还是调封装好的API，该API会以非阻塞的方式调用真正的API，如果返回EAGAIN，那么就往epoll里订阅一个事件，并且通过协程调度器切换到下一个协程，**等到条件满足的时候，epoll就会修改协程为可执行状态，等到这个协程被调度执行的时候，再调一次API就可以成功操作资源了**。

综上, 我们要声明Promise class, 定义协程的一些状态变量。
```cpp
struct Task {
    struct promise_type {
        Task get_return_object() { return {}; }
        std::suspend_never initial_suspend() { return {}; }
        std::suspend_never final_suspend() noexcept { return {}; }
        void return_void() {}
        void unhandled_exception() {}
    }
};
```
get_return_object表示返回为Task对象, Task对象包含对可能返回结果的处理

return_void定义返回为void的处理方法; unhandled_exception()定义异常处理方法

协程再调用co_await内容前会调用initial_suspend(), initial_suspend()可以返回suspend_always或者std::suspend_never, 分别表示协程懒启动lazily-started和紧急启动eagerly-started, 顾名思义, 前者表示只有需要时才启动执行协程。

final_suspend()和initial_suspend()类似, 只不过协程执行完再调用。

这个写法是模板化的，基本所有情况下大概都这样写。

#### co_await
Promise只是定义协程的全局状态, 而对于关键字, 例如co_await。我们直到要暂停当前函数执行, 转而执行新函数。但我们还需要对新函数进行一个规定, 不像go那样不用管了。

以co_await为例, co_await关键字后面跟着的不是一个函数, 而是一个对象, 它规定了co_await的执行逻辑。
```cpp
void AddByCallback(int init, call_back f) //add之后的结果由回调函数f通知
{
    std::thread t([init, f]() { // 开一个线程, 执行逻辑init+100, 然后调用回调函数f
        std::this_thread::sleep_for(std::chrono::seconds(5)); // sleep一下，假装很耗时
        f(init + 100); // 耗时的计算完成了，调用回调函数
    });
    t.detach();
}
Task AddByCoroutine(int init, call_back f)
{
	int ret = co_await AddAWaitable(init);
	//ret = co_await Add100AWaitable(ret);
	//ret = co_await Add100AWaitable(ret);
	f(ret);
}
struct AddAWaitable
{
	AddAWaitable(int init):init_(init) {}
	bool await_ready() const { return false; }
	int await_resume() { return result_; }
	void await_suspend(std::coroutine_handle<> handle)
	{
		auto f = [handle, this](int value) mutable {
			result_ = value;
			handle.resume();
		};
		AddByCallback(init_, f); // 调用原来的异步调用
	}
	int init_;  // 传入的参数
	int result_;    // 返回结果
};

// 调用AddByCoroutine
AddByCoroutine(10, [](int value){ std::cout<<"get result from coroutine1: "<<value<<"\n"; });
```
如上, co_await后面跟的是AddAWaitable对象, 该对象init_成员变量表示传入的参数, 而result_表示协程执行的结果。co_await暂停执行AddByCoroutine(), 转而执行AddAWaitable对象。

AddAWaitable要实现三个函数(规定), await_ready() 用来判断是否执行完毕; await_suspend是AddAWaitable执行的逻辑; await_resume()用来返回值, 也就是`int ret = co_await AddAWaitable(init);` co_await的返回值。

一般的, 协程执行awaiter.await_ready(), 如果返回false则执行await_suspend(), 最后执行await_resume()返回result()。

await_suspend()传入的参数handle, 使用`handle.resume()`用于恢复await处的协程执行。和异步程序相比, 协程逻辑也开了线程执行逻辑, 但区别在于线程执行完的回调函数利用协程上下文直接返回到await执行处, 这个相比于子线程通信十分优雅。但这里协程实际是通过存储和恢复上下文的形式完成信息传递, 让信息传递变得优雅, 而协程本身, 是由于内部的线程池执行的(似乎比自己开效率高些)。类似goroutine内部原理。协程的更多意义可能在于, 不开线程的前提下多给出了一种解决方案。

以下代码可以在https://coliru.stacked-crooked.com/执行, 输入命令`g++ -std=c++20 -O2 -Wall -pedantic -pthread main.cpp && ./a.out`
```cpp
#include <iostream>
#include <thread>
#include <coroutine>
#include <chrono>
#include <functional>

using call_back = std::function<void(int)>;
void AddByCallback(int init, call_back f) // 异步调用
{
	std::thread t([init, f]() {
		//std::this_thread::sleep_for(std::chrono::seconds(5));
		f(init + 100);
	});
	t.detach();
}

struct AddAWaitable
{
	AddAWaitable(int init):init_(init) {}
	bool await_ready() const { return false; }
	int await_resume() { return result_; }
	void await_suspend(std::coroutine_handle<> handle)
	{
		auto f = [handle, this](int value) mutable {
			result_ = value;
			handle.resume();
		};
		AddByCallback(init_, f); // 调用原来的异步调用
	}
	int init_;
	int result_;
};

struct Task // 作为协程的返回值, 这里即AddByCoroutine()函数
{
	struct promise_type {
		auto get_return_object() { return Task{}; }
		auto initial_suspend() { return std::suspend_never{}; }
		auto final_suspend() noexcept { return std::suspend_never{}; }
		void unhandled_exception() { std::terminate(); }
		void return_void() {}
	};
};

Task AddByCoroutine(int init, call_back f)
{
	int ret = co_await AddAWaitable(init);
	ret = co_await AddAWaitable(ret);
	f(ret);
}

int main()
{
    std::cout << "start"<<"\n";
	AddByCallback(5, [](int value){ std::cout<<"get result: "<<value<<"\n"; });
	AddByCoroutine(10, [](int value){ std::cout<<"get result from coroutine1: "<<value<<"\n"; });
	std::cout << "end" <<"\n";
	std::this_thread::sleep_for(std::chrono::seconds(5));
}

输出
start
end
get result: 105
get result from coroutine1: 210
```
由上可见, AddByCoroutine()由于内部声明了`co_await`, 被协程执行, main()线程直接跳过, 直接执行end。而`get result: 105`结果是协程打印的, 不是main线程。这样就实现了异步逻辑调用且十分优雅.协程真滴666!!!

#### co_yield和co_return

以上说明co_wait 协程执行转入`AWaitable`对象, 此外我们还要设置Promise来设置协程启动, 关闭等状态变量, Promise是模板化的可以照抄.

还有两个关键字, co_yield暂停当前协程执行并返回值, 它相当于`co_await promise.yield_value(expr)`.co_yield目的类似python的生成器, 可以讲一个函数暂停(例如循环), 之后再来执行.例如
```cpp
generator<int> Sort(vector<int>& v)
{
    for(int i = int(v.size()) - 1; i >= 0; --i)
    {
        co_yield i;  

    }
}

// 给generator类加上iterator的接口后的用法示例
auto s = Sort(my_vector);
auto it = s.begin();
if(it != s.end())   // 迭代处理迭代器
{
    int j = *it; // 取得本次j的值
    cout << j << " ";
}
```

注意这里实现了一个返回类型为为`generator<int>`的函数, 这个generator和co_await的Task一个作用, Task是co_await所在函数的返回类型, 而generator是co_yield所在函数的返回类型.例如我们在Task对象中写了`promise_type`, 其实Task其他成员编译器已经帮忙写好, 但是`generator<T>`并没有给出, 只能自己写.因此co_yield在C++20基本处于不能用的状态, 以至于cppreference都没有什么定义, 而cpp的生成器功能据说23能出来, 在此不赘述.

co_return直接终止协程返回值, 例如`co_return 7;`, 不赘述.

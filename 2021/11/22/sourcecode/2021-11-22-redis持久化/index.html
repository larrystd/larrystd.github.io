Redis持久化和日志

2021.11.22 larrystd source code  热度 ℃

redis源码参考redis 5.0.14, redis持久化的办法主要是rdb和aof

redis DB

redis DB可以看成若干字典存储的键值对存储, 持久化内容包括键值对和其他一些描述信息。

DB 内部的主要对象是dict字典, 包括键值对字典, 过期键字典。redis本身设计成一个缓存, 每个键会设置一个期限, 超出期限就会过期删除。持久化不会持久化过期的键。此外当内存不足时, redis会基于LRU, LFU等淘汰策略将键值对删除。

redis rio

redis持久化必然要基于IO进行磁盘读写, 因此redis封装了rio来干这些事情。

redis RDB

RDB持久化的方式就是将redis db的内容持久化到文件系统中。

RDB的核心函数时rdbsave和rdbload。rdbsave执行步骤主要有,

  1. 创建持久化文件, 命名temp-$pid.rdb
  2. 向文件写入REDIS$version版本信息
    以下每个db都会存储一列
  3. 保存db标志RDB_OPCODE_SELECTDB, 持久化过期时间等信息
  4. 遍历db中dict迭代器, 持久化db dict的的type, 以string形式持久化dict的key, 以object的形式持久化dict的value。
  5. 最后持久化lua脚本, 间隔标志EOF, 校验和

RDB中的核心思路是Copy-on-Write，来保证在进行快照操作的这段时间，需要压缩写入磁盘上的数据在内存中不会发生变化。在正常的快照操作中，一方面Redis主进程会fork一个快照进程执行RDB操作，另一方面这段时间发生的写操作造成的数据变化会以副本的方式存放在一个新的内存区域，待快照操作结束后才会同步到原来的内存区域, 在此期间可以读取老数据。

触发RDB

用户执行命令save或者bgsave, 将会触发RDB。可见RDB更多是一种手动备份, 但可以设置save 900 1 #900s检查一次，增量的数据变更命令超过1，就触发；自动触发。

server执行shutdown命令关闭server, 关闭之前会进行rdb持久化

AOF

Redis是写后日志，Redis先执行命令，把数据写入内存，然后才记录日志。日志里记录的是Redis收到的每一条命令，这些命令是以文本形式保存。PS: 大多数的数据库采用的是写前日志(WAL)，例如MySQL，通过写前日志和两阶段提交，实现数据和逻辑的一致性。

和RDB直接持久化db的内容不同, AOF开始时, server对执行的命令同时会存储到buf中, 并定期将其buf中的命令进行持久化。此外redis还会定期对命令重写, 即通过创建一个新的AOF文件来替代现有的AOF文件，新旧两个文件所保存的数据库状态是相同的，但新的AOF存储是精简后的命令, 例如将push 1; push 2改成push 1,2, 因此体积会较旧AOF文件小很多。AOF日志也是写后日志, 但可以降低redis宕机之后的数据损失。

AOF日志记录Redis的每个写命令，步骤分为，命令追加（append）、文件写入（write）和文件同步（sync）。

  1. 命令追加, 当AOF持久化功能打开了，服务器在执行完一个写命令之后，会以协议格式将被执行的写命令追加到服务器的 aof_buf 缓冲区。
  2. 文件写入和同步

AOF的调用

之前讲过在eventloop循环之前调用beforeSleep来讲client->buf的数据发送给客户端, 除此之外, beforeSleep期间还会执行依次写aof的操作

之前讲过server的定时器, 事实上server只执行一种定时器且是每个多少秒就会执行依次的, 也就是serverCron。持久化AOF任务包含在定时器中, 因此也是周期执行的。

最后类似rdb, 当server执行shutdown时也会调用AOF持久化。

Redis 4.0 中提出了一个混合使用 AOF 日志和内存快照的方法。简单来说，内存快照以一定的频率执行，在两次快照之间，使用 AOF 日志记录这期间的所有命令操作。

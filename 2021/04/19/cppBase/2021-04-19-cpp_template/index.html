    函数模板

    以下是一个函数模板，返回两个数之间的最大值。T是类型参数。

    函数模板一般不用显示输入类型参数, 自动推导, 类型模板一般不可以

    两阶段编译检查

    模板分两步检查，

    1. 模板定义阶段，只进行类,函数是否定义的检查, 不包含模板类型和对象参数的检查。
    2. 模板实例化阶段，模板会再次被检查，尤其是依赖于类型参数的形参。

    以上可以总结为，和模板参数有关的，总是在第二阶段被检查。

    类型推断

    1. 调用参数按引用传递，任何类型转换都不允许，传入的数组类型引用也不会退化, 通过模板类型T定义的两个参数，它们的实参必须完全一样。
    2. 调用参数按值传递，引用和指针会退化，const和volatile限制符会被忽略。引用转化成被引用的类型，raw array和函数被转换成相应的指针类型。T定义的两个参数，实参退化后类型必须一样。

    有两种办法解决推导歧义的错误

    返回类型推断

    C++11 中，尾值返回类型(trailing reture type) 用箭头表示声明返回类型。可以使用decltype推导表达式, decltype可能也是基于语法树推导返回类型, 推导表达式不要太复杂

    需要注意，T1可能是引用因此会推导出返回值为引用，但我们想让它返回退化后的T

    struct declay, Applies lvalue-to-rvalue, array-to-pointer, and function-to-pointer implicit conversions to the type T, removes cv-qualifiers, and defines the resulting type as the member typedef type, 数组会推导成指针。

    继承

    对于类模板，如果它的基类也依赖于模板参数，那么this->x和x不一定等效。

    以上Derived中的bar永远不会被解析成Base中的bar,作为经验法则，建议用this->或base<T>::修饰之。

    值传递

    当按值传递参数时，原则上所有的参数都会被拷贝，但注意被传递的参数是左值lvalue，拷贝构造函数会被调用。但是传递参数是右值，编译器会优化参数传递，使拷贝构造函数不会被调用。

    按值传递会导致类型退化decay，裸数组退化成指针，const和volatile等限制符被解除。

    引用传递

    引用传递类型不会退化，也就是裸数组不会转换成指针，也不会移除const和volatile等限制符。

    按转发引用传递参数

    综上，左值与通用引用放在一推导出来的T仍为左值，而右值与通用引用放在一起推导出来的T仍然为右值

    虚函数不能设置成模板函数

    同理，虚函数不能设置为模板函数(但可以使用类的模板)，因为对于虚函数来说, 它调用多少次需要运行时确定。因为Base* base = new Drive;条件下base会调用派生类的虚函数。但对于模板来说, 它必须编译时确定函数被调用多少次, 并据此实例化几个不同类型的函数。既然不能确定虚函数调用多少次，因此虚函数不能设置模板。

    另外对于虚函数表来说，它的大小是固定的，换言之一个class对应一个虚函数表, 对于Base* base = new Drive;, base使用Drive class的虚函数表, 因此对于虚函数它会调用Derive的函数, 对非虚函数它还是使用Derive的表, 因为Derive对象的非虚函数继承自Base。但是如果虚函数是模板, 那岂不是模板类型为int实例化一个, float实例化一个, 虚函数表大小会大大膨胀, 这也不应该。

    构造函数可以是模板参数, 析构函数没有参数, 因此也不存在模板情况。

    模板继承问题

    这里涉及到模板的两阶段查找问题Two phase lookup, 准确的说

    1. 在模板定义阶段，先对独立的(与模板参数无关的)成员进行检查，检查不通过则编译报错;
    2. 在实例化模板时，对非独立的成员检查(检查是否在子类或基类中)

    这样对于继承, 第一阶段查找会把父类和子类分别查找, 因此子类调用父类函数就可能编译出错(因为它不关心父类)。但如果第一阶段编译通过，第二阶段就把模板类作为非独立的检查一遍，这时候就会通过。

    似乎模板修饰class存在三个主要区别, 继承由于第一阶段检查可能编译不过, 虚函数不能设置为模板参数函数, 不同编译单元的模板可能延迟编译导致无法链接。其他无太大区别。

    类似的还有inline函数, 一般inline函数必须把定义放在.h文件中, 目的是所有用到inline函数的编译单元都有inline函数的定义, 而不是通过链接(inline函数必须编译在编译单元中因为它不基于函数调用)。链接意味着调用别的编译单元的函数必须要进行调用, 另外如果类成员函数在.h中定义, 那它也是inline的, 因此短的类成员函数也可以在.h中定义比较函数调用的损耗。

    总结

    函数模板自动推导参数, 类模板一般不行。

    两阶段检查, 第一阶段不会检查参数, 继承依赖关系等。

    类型推断, 模板参数值传递会数组退化, 去掉指针引用修饰符, const被忽略; 引用传递则保持原来的类型,数组也不会退化。

    模板默认构造参数, 返回类型推断

    模板类特例化, 灵活使用模板传参

    typename声明类型, 不会产生对象创建歧义。

    模板可以传入多种类型, 某种意义上可以是静态多态。静态多态只要通过编译运行时是安全的

    模板多个编译单元链接时可能存在问题, 同样两个模板类有继承依赖关系时也可能存在问题

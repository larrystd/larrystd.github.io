### 并发编程

并发编程有两种基本模型,message passing消息传递和shared memory共享内存。
线程同步的四项原则，按重要性排序:
* 首要原则是最低限度共享对象。一个对象能不暴露给别的线程就不暴露，如果暴露尽量选择immutable对象。
* 其次使用高级并发组件，线程池、队列、倒计时，如TaskQueue, CountDownLatch等
* 最后不得已使用底层同步原语，**只用不可重入互斥器和条件变量，慎用读写锁，不用信号量**。
* 除了使用atomic整数，不不编写lock-free代码，不用内核级同步原语。

底层同步原语一般包括互斥器mutex, 条件变量condition variable。互斥器和条件变量构成了多线程编程的全部必备同步原语。

#### 互斥器mutex
mutex作用是保护临界区，任何时刻最多只有一个线程在临界区内活动。基本原则是
* 用RAII手法封装mutex的创建、销毁、加锁、解锁四个操作
* 只使用非递归的mutex(即不可重入的mutex)
* 不手动调用lock()和unlock()函数，一切交给栈上的Guard对象的构造和析构函数负责。这样**始终在同一个函数的同一个scope里对某个mutex进行加锁或者解锁**，如果出现死锁Scoped Locking很容易定位。
* 每次构造Guard对象时，思考一路调用栈上持有的锁，防止因加锁顺序不同导致死锁。
* 进程间通信只用tcp socket，不用跨进程/函数的mutex。

#### 只用非递归的mutex
mutex分为递归recusive和非递归non-recusive两种。考虑以下代码
```cpp
MutexLock mutex;
std::vector<Foo> foos;

void post(const Foo& f) {
    MutexLockGuard lock(mutex);
    foos.push_back(f);
}

void traverse() {
    MutexLockGuard lock(mutex);
    for (std::vector<Foo>::const_iterator it = foos.begin(); it != foos.end(); ++it) {
        it->doit();
    }
}

```
注意post和traverse都是操作同一个mutex，post加锁，修改foos对象; traverse() 加锁，然后遍历foos向量。

如果`Foo::doit()`间接调用了`post()`，那么
* mutex是非递归的，循环请求mutex,将会死锁
* mutex是递归的, 将一直执行push_back函数，可能导致迭代器失效。

非递归mutex的优越性是死锁比较容易debug，轻易把程序逻辑错误暴露出来，把各个线程的调用栈打出来即可。（感觉用迭代器失效证明非递归锁不好的例子不太好）

### 条件变量 condition variable
在使用mutex时候，我们希望加锁不要阻塞，总能立刻拿到锁，尽快访问完数据，尽快解锁。条件变量用来等待某个条件成立。

对于wait端，必须与mutex一起使用，在mutex上锁条件下才能调用wait，**把判断条件和wait()放到while循环中**。释放锁进入等待。

条件变量是非常底层的同步原语，一般都是用来实现高层的同步措施，例如`BlockingQueue<T>`或`CountDownLatch`。

倒计时CountDownLatch是一种常用的同步手段，主要有两种用途:
* 主线程发起多个子线程，等这些子线程都完成一定任务之后，主线程才继续执行。即主线程等待子线程完成初始化。
* 子线程等待主线程，主线程完成其他一些任务之后通知所有子线程开始执行。

```cpp
class CountDownLatch : boost::noncopyable {
public:
    explicit CountDownLatch(int count); // 倒数几次
    void wait();    //等待计数值变为0
    void countDown();   // 计数减1
private:
    mutable MutexLock mutex_;
    Condition condition_;
    int count_;
};

void CountDownLatch::wait() {
    MutexLockGuard lock(mutex_);
    while (count_ > 0)
        condition_.wait();
}

void CountDownLatch::countDown() {
    MutexLockGuard lock(mutex_);
    --count_;
    if (count_ == 0)
        condition_.notifyAll();
}
```
### 封装MutexLock, MutexLockGuard, Condtion

如下
```cpp
class MutexLock : boost::noncopyable{
public:
    MutexLock() : holder_(0) {
        pthread_mutex_init(&mutex_, NULL);
    }
    ~MutexLock() {
        assert(holder_ == 0);
        pthread_mutex_destroy(&mutex_)
    }

    bool isLockedByThisThread() {
        return holder_ == CurrentThread::tid();
    }

    void assertLocked() {
        assert(isLockedByThisThread());
    }
    void lock() {
        pthread_mutex_lock(&mutex_);
        holder_ = CurrentThread::tid();
    }

    void unlock() {
        holder_ = 0;
        pthread_mutex_unlock(&mutex_);
    }

    pthread_mutex_t* getThreadMutex() {
        return &mutex;
    }
private:
    pthread_mutex_t mutex_;
    pid_t holder_;
};

class MutexGuard : boost::noncopyable {
public:
    explicit MutexLockGuard(MutexLock& mutex)
        : mutex_(mutex)
    { mutex_.lock(); }

    ~MutexLockGuard() 
    { mutex_.unlock(); }

private:
    MutexLock& mutex;
};

class Condition : boost::noncopyable {
public:
    explicit Condition(MutexLock& mutex)
        : mutex_(mutex)
    {   pthread_cond_init(&pcond_, NULL); }

    ~Condition() {
        pthread_cond_destroy(&pcond_);
    }

    void wait() {
        pthread_cond_wait(&pcond_, mutex_.getPthreadMutex());
    }

    void notify() {
        pthread_cond_signal(&pcond_);
    }

    void notifyAll() {
        pthread_cond_boardcast(&pcond_);
    }
private:
    MutexLock& mutex_;
    pthread_cond_t pcond_;
}
```

注意，在同时使用mutex和condtion时,mutex应该先于condition构造，并作为后者的构造参数。
```cpp
class CountDownLatch {
public:
    CountDownLatch(int count)
        : mutex_(),
        condition_ (mutex),
        count_(count)
    {   }
private:
    mutable MutexLock mutex_;
    Condtion condtion_;
    int count_;
};
```

等待事件发生，正确的做法是用`select()`或者`Condition`，不能用sleep来进行轮循。

### 借助shared_ptr实现copy-on-write，mutex取代读写锁

copy-on-write目的是降低copy带来的效率降低, 表现在需要的时候再copy, 只copy需要的部分。例如fork一个进程, 子进程不是开始就copy父进程内存到自己的内存空间, 而是先共享父进程之内存空间, 等待需要写时再按需给子进程开辟对应的空间。

copy-on-write的实现需要考虑, 并发读时需要判断有无人在写, 写时需要判断有没有人在读写。如果有, 则需要copy; 没有则可以直接写。

如下代码
```cpp
MutexLock mutex;
std::vector<Foo> foos;

void post(const Foo& f) {
    MutexLockGuard lock(mutex);
    foos.push_back(f);
}

void traverse() {
    MutexLockGuard lock(mutex);
    for (std::vector<Foo>::const_iterator it = foos.begin(); it != foos.end(); ++it) {
        it->doit();
    }
}
```

以上是一个读写逻辑，其中`post()`为写端，`traverse()`为读端。可以发现读和写都需要加锁。这种没有必要, 因为多个线程并发读是不需要加锁的。特别是在读多写少的场景中, 这种方法并不经济。此外, 如果`it->doit();`语句调用`post`将会发生死锁。


用`shared_ptr`来管理共享数据(对象)原理如下
* shared_ptr是引用计数型智能指针，初始引用计数为1.
* 对于write端，如果发现引用计数为1,可安全修改共享对象而不必担心有人在读; 如果引用计数>1，利用copy-on-write生成副本并修改副本。
* 对于read端，读之前引用计数+1, 读之后引用计数-1，这样读期间引用计数大于1,可以阻止并发写。

多个线程并发读, 可以正常读, 不用加锁。但是将g_foos赋值给foos使引用计数+1这个过程需要加锁, 但`foos = g_foos`时间远小于`foos.push_back(f);`, 因此这降低了锁的粒度, 增加了效率。

写操作需要加锁, 期间禁止其他线程读。但注意可能有线程在执行`it->doit()`, 因此写操作还得判断引用计数, 引用计数>1说明有线程在执行`it->doit();`, 写操作复制写, 

```cpp
typedef std::vector<Foo> FooList;
typedef boost::shared_ptr<FooList> FooListPtr;
MutexLock mutex;
FooListPtr g_foos;  // 用来观察的shared_ptr,read计数+1，write看计数值

// read端
void traverse() {
    FooListPtr foos;
    {
        MutexLockGuard lock(mutex);
        foos = g_foos;  // read端，使g_foos对象引用计数+1。这时候foos, g_foos引用计数为2
        assert(!g_foos.unique());   // 独占对象，use_count=1
    }
    for (std::vector<Foo>::const_iterator it = foos.begin(); it != foos.end(); ++it) {
        it->doit();
    }
    // foos销毁，g_foos计数-1
}

// write端
void post(const Foo& f) {
    printf("post\n");
    MutexLockGuard lock(mutex);
    if (!g_foos.unique()) {// 计数不为1,说明有在读，
        g_foos.reset(new FooList(*g_foos)); // 创建副本,g_foos复制成新的std::vector<Foo>。这里巧妙的是, reset让old list的引用计数-1, 如果再读完, old list将自动析构。之后用到的都是新的list
        printf("copy the whole list\n");
        // ...
    }
    assert(g_foos.unique());    // 如果g_foos引用计数>1, 放弃
    g_foos->push_back(f);   // f加入std::vector<Foo>中
}
```
以上，通过`mutex`和`shared_ptr`的引用计数，可以用`mutex`实现读写锁，性能比后者高。
